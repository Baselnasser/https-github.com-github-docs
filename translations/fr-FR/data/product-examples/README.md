# Exemples d’atterrissage de produit

Les pages qui utilisent la disposition `product-landing` peuvent éventuellement inclure une section `Examples`. Nous prenons en charge trois types d’exemple pour le moment :

1. Exemples de code. Consultez https://docs.github.com/en/codespaces#code-examples.

2. Exemples provenant de la communauté. Consultez https://docs.github.com/en/discussions#community-examples.

3. Exemples utilisateur. Consultez https://docs.github.com/en/sponsors#community-examples.

## Fonctionnement

Des exemples de données pour chaque produit sont définis dans `data/product-landing-examples`, dans un sous-répertoire nommé pour le **produit** et un fichier YML nommé pour le **type d’exemple** (par exemple `data/product-examples/sponsors/user-examples.yml` ou `data/product-examples/codespaces/code-examples.yml`). Nous ne prenons en charge qu’un seul type d’exemple par produit.

### Gestion de version

Pour le moment, la gestion de versions est prise uniquement en charge dans les exemples de code. Si un exemple de bloc doit être disponible dans **toutes** les versions, vous n’avez rien de spécial à faire. Toutefois, si un exemple de bloc doit être disponible uniquement dans certaines versions, vous pouvez ajouter une propriété `versions` comme ceci :

```
- title: Dependabot version update PR
  description: Example pull request generated by the Dependabot version updates configuration in the Super linter repository.
  href: /github/super-linter/pull/1398
  languages:
  tags:
    - Dependabot
    - Version updates
    - Pull requests
  versions:
    fpt: '*'
  ```

où la syntaxe de `versions` est la même que celle de la [propriété `versions` des informations préliminaires](content/README.md), et peut prendre en charge la notation SemVer.

## Rendu

Les exemples de données de produit sont ajoutés à l’objet `context` dans `middleware/contextualizers/product-examples.js`.

Les données sont ensuite affichées par `components/landing`.
