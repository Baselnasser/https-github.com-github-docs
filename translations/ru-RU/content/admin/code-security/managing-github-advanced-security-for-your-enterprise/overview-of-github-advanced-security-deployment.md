{% data variables.product.prodname_GH_advanced_security %} (GHAS) позволяет командам разработчиков писать более безопасные коды с помощью интегрированных инструментов, включая CodeQL — самый сложный механизм анализа семантического кода в мире. GHAS — это набор средств, требующих активного использования со стороны разработчиков в организации. Чтобы добиться максимальной рентабельности инвестиций и надежной защиты кодов, специалисты вашей организации должны научиться использовать и применять средства GHAS, а также обеспечивать их надлежащее обслуживание.

Одни из самых сложных задач, с которыми сталкиваются компании при приобретении нового программного обеспечения, — это его развертывание и реализация, а также внедрение культурных изменений, которые могли бы повысить заинтересованность в масштабах всей организации и обеспечить успешное развертывание ПО.

Мы подготовили этот обзор, чтобы помочь вам понять эти процессы и подготовиться к ним, используя инструменты GHAS. Этот обзор:
  - позволит составить представление о том, как развертывание GHAS может быть реализовано в вашей компании;
  - поможет подготовить вашу компанию к развертыванию;
  - предоставит основные рекомендации по повышению шансов на успешное развертывание.

Сведения о возможностях обеспечения безопасности, доступных в {% data variables.product.prodname_GH_advanced_security %}, см. в разделе "[Функции безопасности {% data variables.product.prodname_dotcom %}](/code-security/getting-started/github-security-features)".

## <a name="recommended-phased-approach-for-ghas-rollouts"></a>Мы рекомендуем поэтапный подход к развертыванию GHAS

Мы создали поэтапный подход к развертыванию GHAS на основе лучших отраслевых методик и рекомендаций GitHub. Вы можете использовать этот подход для развертывания с помощью {% data variables.product.prodname_professional_services %} или же независимо.

Мы рекомендуем подходить к развертыванию поэтапно, однако вы можете вносить корректировки по ходу процесса с учетом потребностей вашей компании. Мы также рекомендуем создать временную шкалу развертывания и реализации и придерживаться ее. На начальном этапе планирования мы можем помочь вам определить идеальный подход и составить временную шкалу, которая лучше всего подходит для вашей компании.

![Схема, на которой показаны три этапа развертывания GitHub Advanced Security. Этап 0: планирование и старт проекта. Этап 1: пилотные проекты. Этап 2: создание вовлеченности и развертывание для ранних последователей. Этап 3: полное развертывание в масштабе организации и управление изменениями](/assets/images/enterprise/security/advanced-security-phased-approach-diagram.png)


Исходя из опыта успешного развертывания {% data variables.product.prodname_GH_advanced_security %} в компаниях наших клиентов, мы ожидаем, что большинство клиентов будут придерживаться этих этапов. Учитывая потребности компании, вы можете менять подход, удаляя или изменяя некоторые этапы и шаги.

Подробные инструкции по реализации каждого этапа см. в статье "[Развертывание {% data variables.product.prodname_GH_advanced_security %} в вашей организации](/admin/advanced-security/deploying-github-advanced-security-in-your-enterprise)". В следующем разделе приведены общие сведения о каждом из этих этапов.

###  <a name="-octicon-milestone-aria-labelthe-milestone-icon--phase-0-planning--kickoff"></a>{% octicon "milestone" aria-label="The milestone icon" %} Этап 0. Планирование и запуск

Общий смысл этого этапа заключается в планировании и подготовке организации к развертыванию. Подразумевается, что у организации есть люди, процессы и технологии, необходимые для успешного развертывания. На этом этапе также следует обсудить, какие критерии будут учитываться при оценке реализации и использования GHAS в пределах организации.

### <a name="-octicon-milestone-aria-labelthe-milestone-icon---phase-1-pilot-projects"></a>{% octicon "milestone" aria-label="The milestone icon" %} Этап 1. Пилотный запуск проекта

Начинать реализацию GHAS мы рекомендуем с нескольких проектов или команд с высоким уровнем влияния, для которых и будет предназначено первоначальное развертывание. Таким образом, начальная группа специалистов сможет ознакомиться с инструментами GHAS, узнать, как включать и настраивать их. Это позволит построить фундамент для развертывания GHAS в масштабе всей организации.

### <a name="-octicon-milestone-aria-labelthe-milestone-icon---phase-2-organizational-buy-in--rollout-preparation"></a>{% octicon "milestone" aria-label="The milestone icon" %}  Этап 2. Как заручиться поддержкой и подготовиться к развертыванию

На этом этапе проводится оценка результатов предыдущих этапов и выполняется подготовка к развертыванию в масштабе компании. Оценка заинтересованности организации после пилотных проектов позволяет принять окончательное решение: будет ли компания продолжать работу в привычном ключе или же перейдет к постепенному внедрению GHAS (чаще всего принимается именно это решение). Если компания решает постепенно внедрять GHAS в свои процессы, то этап 2 переходит в этап 3 и т. д.

### <a name="-octicon-milestone-aria-labelthe-milestone-icon---phase-3-full-organizational-rollout--change-management"></a>{% octicon "milestone" aria-label="The milestone icon" %} Этап 3. Полное управление развертыванием и изменениями в организации

После согласования решения ваша компания может приступить к развертыванию GHAS в масштабе всей организации в соответствии с планом развертывания. На этом этапе важно убедиться, что план учитывает любые организационные изменения, которые могут потребоваться во время развертывания GHAS, и что ваши команды понимают необходимость, ценность и влияние изменений на текущие рабочие процессы.

## <a name="best-practices-for-a-successful-ghas-rollout"></a>Рекомендации по успешному развертыванию GHAS

Мы выяснили, что компании, которые успешно завершили развертывание GHAS, имеют схожие качества, которые и помогли им сделать это. Чтобы повысить шансы вашей компании на успешное развертывание GHAS, ознакомьтесь с этими рекомендациями.

### <a name="-octicon-checklist-aria-labelthe-checklist-icon--set-clear-goals-for-your-companys-rollout"></a>{% octicon "checklist" aria-label="The checklist icon" %} Установка четких целей развертывания

Казалось бы, постановка целей — это очевидный этап. Однако мы часто наблюдаем, как компании приступают к развертыванию GHAS без каких-либо четких целей. Таким компаниям сложнее создать истинную заинтересованность, без которой не будет ни успешного развертывания, ни реализации ценности GHAS.

Приступая к планированию и реализации GHAS, начните с постановки целей и убедитесь, что эти цели понятны и очевидны для ваших специалистов. Не обязательно прорисовывать цели во всех подробностях — они могут быть простыми. Главное, чтобы у вас была точка отсчета и чтобы эти цели были согласованы. Это поможет создать основу и задать направление развертывания, а также разработать план, который поможет вашей компании достичь конечной точки. Если вам требуется помощь в постановке целей, специалисты {% data variables.product.prodname_professional_services %} дадут вам рекомендации, основанные на нашем опыте работы с вашей компанией и другими клиентами.

Вот несколько общих целей развертывания GHAS, которые вы можете использовать как примеры.
  - **Сокращение количества уязвимостей.** Эта цель может быть универсальной или может быть связана с недавним инцидентом, который, как вам кажется, ваша компания могла бы предотвратить с помощью GHAS.
  - **Определение репозиториев с высоким риском.** Иногда требуется просто определить репозитории, которые наиболее подвержены рискам, чтобы начать устранять уязвимости и сокращать эти риски.
  -  **Увеличение частоты исправлений.** Этого можно добиться путем внедрения результатов и своевременного устранения уязвимостей, чтобы предотвратить накопление проблем безопасности.  
  - **Соответствие нормативным требованиям.** Эта цель может подразумевать только создание новых требований к обеспечению соответствия или же что-то более конкретное. Мы отмечаем, как многие медицинские компании используют GHAS, чтобы предотвратить раскрытие персональных медицинских данных пациентов.
  - **Предотвращение утечки секретов.** Часто такую цель ставят перед собой компании, которые уже сталкивались с утечкой критически важных данных (ключей программного обеспечения, финансовых или персональных данных и т. д.) или хотели бы предотвратить эту проблему.
  - **Управление зависимостями.** Такую цель часто преследуют компании, которые пострадали от взломов, связанных с неисправленными зависимостями, и компании, которые хотят предотвратить подобные атаки за счет обновления уязвимых зависимостей.  

### <a name="-octicon-checklist-aria-labelthe-checklist-icon--establish-clear-communication-and-alignment-between-your-teams"></a>{% octicon "checklist" aria-label="The checklist icon" %} Взаимодействие между командами и согласованность их действий

Четкое взаимодействие и согласование действий имеют решающее значение для успешного выполнения любого проекта, и развертывание GHAS не исключение. По нашим наблюдениям, компании, где специалисты по обеспечению безопасности тесно взаимодействуют с разработчиками и исполнительными лицами (руководителем по информационной безопасности или вице-президентом) на всех этапах — от приобретения GHAS до непосредственного развертывания, — имеют больше шансов на успешное развертывание GHAS.

Помимо обеспечения согласованности действий этих групп во время развертывания GHAS, мы также рекомендуем сосредоточиться на следующих аспектах и вопросах.

#### <a name="rollout-planning"></a>Планирование развертывания

Как вы будете развертывать GHAS в вашей компании? Здесь может быть множество идей и мнений. Ответьте на следующие вопросы, перед тем как приступить к развертыванию.
  - Какие команды будут участвовать в пилотном проекте?
  - На какие проекты будет ориентирован пилотный проект?
  - Каким образом будут назначаться приоритеты для проектов в рамках развертывания?
  - Каким образом будет оцениваться успешность пилотного (и не только) проекта?
  - К какому объему ежедневных изменений готовы ваши специалисты? Каким образом они будут узнавать об этом?
  - Как вы планируете информировать сотрудников о планах на развертывание в масштабе всей компании?
  - Как вы планируете обучать команды?
  - Каким образом вы будете управлять результатами сканирования изначально? (Дополнительные сведения см. в следующем разделе, посвященном обработке результатов.)

#### <a name="processing-results"></a>Обработка результатов

Перед развертыванием GHAS в ваших командах необходимо четко обозначить, как со временем будет осуществляться управление результатами. В первое время мы рекомендуем не применять к результатам никаких блокировок — это повышает их информативность. Скорее всего, в вашей компании функционирует конвейер непрерывной поставки и непрерывной интеграции (CI/CD), поэтому мы рекомендуем придерживаться такого подхода, чтобы избежать блокировки процессов компании. Когда подобная обработка результатов войдет в привычку, вы сможете постепенно увеличивать объем ограничений в соответствии с потребностями компании.

### <a name="-octicon-checklist-aria-labelthe-checklist-icon---lead-your-rollout-with-both-your-security-and-development-groups"></a>{% octicon "checklist" aria-label="The checklist icon" %} Привлечение к развертыванию специалистов по обеспечению безопасности и разработчиков

Многие компании ведут работу по развертыванию GHAS с привлечением группы безопасности. Зачастую разработчики не принимают участие в развертывании до завершения пилотного проекта. Тем не менее, мы выяснили, что компании, которые привлекают к работе и группы безопасности, и разработчиков, как правило, имеют больше шансов на успешное развертывание GHAS.

Почему? Подход к обеспечению безопасности программного обеспечения, который используется в GHAS, ориентирован на разработчиков и без труда интегрируется в их рабочие процессы. Исключение группы разработки на ранних этапах процесса развертывания повышает риски и затрудняет создание должного уровня заинтересованности в GHAS в масштабе компании.

Если разработчики включены в процесс на начальном этапе (в идеале с момента покупки), то группы безопасности и разработки могут быстро достичь согласованности действий. Это помогает устранить разрозненность между группами и укрепить рабочие отношения между ними. Все это обеспечивает лучшее взаимодействие между специалистами и не дает им перекидывать проект из рук в руки. Все эти рекомендации направлены на достижение общей цели: помочь компании внедрить средства GHAS и начать их использовать для устранения проблем безопасности на самых ранних этапах разработки.

#### <a name="-octicon-people-aria-labelthe-people-icon--recommended-key-roles-for-your-rollout-team"></a>{% octicon "people" aria-label="The people icon" %} Рекомендуемые участники команды развертывания

Мы рекомендуем включить в команду несколько ключевых лиц, которые будут представлять все уровни компании на всех этапах проекта: от планирования до развертывания и реализации.

Мы настоятельно рекомендуем, чтобы в вашей команде по развертыванию были представлены следующие роли.
- **Исполнительный спонсор.** Чаще всего это руководитель по информационной безопасности, директор по ИТ, вице-президент по безопасности или проектированию.
- **Ведущий технический специалист по безопасности.** Этот специалист обеспечивает техническую поддержку проекта от лица команды по обеспечению безопасности на всех этапах реализации.
- **Ведущий технический специалист по разработке.** Этот специалист обеспечивает техническую поддержку и участвует в реализации от лица команды разработчиков.  

Мы также рекомендуем включить в команду по развертыванию следующих специалистов.
- **Руководитель проекта.** Мы полагаем, что чем раньше руководитель проекта будет привлечен к реализации, тем выше вероятность успешного развертывания.  
- **Инженер по контролю качества.** Привлечение этого специалиста позволит команде по контролю качества быть в курсе всех изменений в процессах и учитывать их в работе.

### <a name="-octicon-checklist-aria-labelthe-checklist-icon--understand-key-ghas-facts-to-prevent-common-misconceptions"></a>{% octicon "checklist" aria-label="The checklist icon" %} Разбор фактов о GHAS для предотвращения распространенных заблуждений

#### <a name="fact-1-ghas-is-a-suite-of-security-tools-that-require-action-to-protect-your-code"></a>Факт 1. GHAS — это набор средств безопасности, которые требуют активных действий для защиты кода.

Это не ПО безопасности, которое можно установить и забыть. Развертывание GHAS само по себе не обеспечивает защиту кода. Ценность GHAS повышается, лишь когда эти средства настраиваются, обслуживаются и используются в ежедневных рабочих процессах наряду с другими инструментами.

#### <a name="fact-2-ghas-will-require-adjustment-out-of-the-box"></a>Факт 2. GHAS требует корректировки после развертывания.

После установки GHAS в репозиториях необходимо выполнить дополнительные действия, чтобы обеспечить работу GHAS в соответствии с потребностями вашей компании. Функция сканирования кода, в частности, требует расширенной настройки результатов, например того, какие события будут помечаться сканированием и что будет выбираться в будущих сканированиях. Многие клиенты обнаруживают, что начальное сканирование либо вообще не приводит к результатам, либо выдает результаты, не соответствующие модели рисков приложения, и требует дополнительной настройки к учетом потребностей компании.

#### <a name="fact-3-ghas-tools-are-most-effective-when-used-together-but-the-most-effective-appsec-programs-involve-the-use-of-additional-toolsactivities"></a>Факт 3. Инструменты GHAS наиболее эффективны при совместном использовании, но самые эффективные программы AppSec предполагают использование дополнительных средств и действий.

GHAS наиболее эффективен, когда все средства используются вместе. Когда компании интегрируют GHAS с другими инструментами и действиями, такими как тестирование на проникновение и динамическое сканирование, это еще больше повышает эффективность программы AppSec. Мы рекомендуем всегда использовать несколько уровней защиты.

#### <a name="fact-4-not-all-companies-will-useneed-custom--data-variablesproductprodname_codeql--queries-but-they-can--help-you-customizetarget-scan-results"></a>Факт 4. Не все компании будут использовать пользовательские запросы или нуждаться в них {% data variables.product.prodname_codeql %}, но они могут помочь настроить результаты сканирования или определить их цель.

Сканирование кода на платформе {% data variables.product.prodname_codeql %} — самой мощной подсистеме анализа кода в мире. Хотя многие компании рады возможности писать пользовательские запросы, для большой части наших клиентов базового набора запросов и дополнительных запросов, доступных в сообществе, как правило, больше, чем достаточно. Тем не менее, многим компания могут потребоваться пользовательские запросы {% data variables.product.prodname_codeql %}  для сокращения числа ложноположительный результатов или для создания новых запросов для целевых результатов, которые могут потребоваться вашей компании.

Однако если ваша компания заинтересована в написании пользовательских запросов {% data variables.product.prodname_codeql %}, рекомендуется завершить развертывание и реализацию GHAS прежде, чем изучать пользовательские запросы.

{% note %}

**Примечание:** Для вашей компании важно обеспечить надежный фундамент на основе GHAS, прежде чем углубляться в более сложные методики безопасности.

{% endnote %}

Когда ваша компания будет готова, наша команда по обеспечению успеха клиентов может помочь вам выполнить необходимые требования и убедиться, что в компании имеются хорошие варианты использования пользовательских запросов.  

#### <a name="fact-5--data-variablesproductprodname_codeql--scans-the-whole-code-base-not-just-the-changes-made-in-a-pull-request"></a>Факт 5. {% data variables.product.prodname_codeql %} сканирует всю базу кода, а не только изменения, сделанные в запросе на вытягивание.

При выполнении сканирования кода из запроса на вытягивание сканирование будет включать полную базу кода, а не только изменения, сделанные в запросе на вытягивание. Хотя иногда это может показаться ненужным, это важный шаг, направленный на то, чтобы убедиться, что изменение было проверено на всех взаимодействиях в базе кода.
